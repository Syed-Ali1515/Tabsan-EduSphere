# Tabsan EduSphere — Database Seed Scripts

Three SQL scripts that take you from **zero to a fully seeded database** with
no toolchain dependencies.  All scripts are **idempotent** — safe to run
multiple times without corrupting data.

---

## Scripts

| # | File | Purpose |
|---|---|---|
| 0 | `0-Schema.sql` | **Run first.** Creates the `TabsanEduSphere` database (if absent) and applies every EF Core migration idempotently — all 49 tables, indexes, views, and stored procedures. |
| 1 | `1-MinimalSeed.sql` | Minimal baseline — roles, all 14 modules, sidebar menus, report definitions, license state, and 4 test users (SuperAdmin / Admin / Faculty / Student). Enough to log in and exercise every screen. |
| 2 | `2-FullDummyData.sql` | Everything in Script 1 **plus** 3 departments, 10 courses, 8 offerings, 3 admins, 5 faculty, 10 students, timetables, assignments, attendance, results, quizzes, FYP projects, notifications, payment receipts, AI-chat, and email logs. Exercises all views and stored procedures. |

## Generating Test-User Password Hashes

All test accounts use the same password.  Because Argon2id hashing cannot be
done in plain SQL, generate a hash first and paste it into the scripts.

### Option A — PowerShell helper (recommended)

```powershell
# From the repository root
.\Scripts\GenerateTestHashes.ps1
```

The script will print an Argon2id hash for the password you choose, then
**automatically patch** `1-MinimalSeed.sql` and `2-FullDummyData.sql` with
that hash.

### Option B — manual

1. Open `1-MinimalSeed.sql` / `2-FullDummyData.sql`.
2. Find the line that reads:

   ```sql
   DECLARE @PwdHash NVARCHAR(500) =
       N'PLACEHOLDER_RUN_GenerateTestHashes.ps1_TO_REPLACE';
   ```

3. Replace the placeholder value with an Argon2id hash generated by any
   compatible tool (memory=65536 KB, iterations=3, parallelism=4, length=32).

## Quick Start (fresh machine)

Run all three scripts in order from the repository root:

```powershell
# Step 1 — create database + full schema (tables, views, stored procs)
sqlcmd -S "(localdb)\mssqllocaldb" -d master -i "Scripts\0-Schema.sql"

# Step 2 — seed test data (choose one)
sqlcmd -S "(localdb)\mssqllocaldb" -d master -i "Scripts\1-MinimalSeed.sql"
# OR for full dummy data:
sqlcmd -S "(localdb)\mssqllocaldb" -d master -i "Scripts\2-FullDummyData.sql"
```

> **No `dotnet ef` or API startup required.** `0-Schema.sql` replaces the
> migration step entirely.

## Running Individual Scripts

Use **SQL Server Management Studio**, **Azure Data Studio**, or `sqlcmd`.  
Always run `0-Schema.sql` first if the database does not exist yet.

## Preflight Checks

### 1 — Verify migration count before seeding

The seed scripts require all 16 EF Core migrations to be applied.  Run this
one-liner to confirm:

```powershell
sqlcmd -S "(localdb)\mssqllocaldb" -d TabsanEduSphere `
  -Q "SELECT COUNT(*) AS MigrationCount FROM __EFMigrationsHistory"
```

**Expected result**: `16`

If the count is 0 or the command errors, apply migrations first:

```powershell
dotnet ef database update `
  --project  src/Tabsan.EduSphere.Infrastructure `
  --startup-project src/Tabsan.EduSphere.API `
  --context  ApplicationDbContext
```

Then re-run the seed script.

### 2 — "Schema missing" message (first-run only)

If you run a seed script against a brand-new, empty database you will see:

```
Schema missing in TabsanEduSphere (table dbo.roles not found).
Database has been created successfully.
Next step: run EF migrations, then run this script again.
```

This is **expected** — the script created the database file but the schema
(tables, views, stored procedures) does not exist yet.  Apply migrations as
shown above, then run the script again.

### 3 — Idempotency

Both scripts are safe to run multiple times.  Every `INSERT` is guarded by a
`NOT EXISTS` check, so no duplicate rows are created.  Running Script 1 after
Script 2 (or vice-versa) will not corrupt data.
